# Assegai ORM 0.9.5 Release Notes

Assegai ORM 0.9.5 hardens EntityManager write paths across inserts, updates, upserts, and soft removes. This release keeps the relation-ID alias work from 0.9.4 intact while tightening validation, restoring safe bulk insert support, improving generated identifier reporting, and making nullable/default conversion behavior more predictable.

## Highlights

- `insert()` supports list-style bulk payloads again, with validation for every row before any write is attempted.
- Bulk inserts now populate generated identifiers for SQLite and MySQL, including mixed batches where some rows provide explicit primary keys and others rely on database generation.
- Relation ID aliases are normalized consistently across update, insert, upsert, readback, null-write, and conflict-path handling.
- Declared entity columns now take precedence over generated relation aliases when names collide.
- Duplicate column writes preserve meaningful zero values and fail clearly when scalar, raw-column, and relation-alias values conflict.
- Partial object writes now skip `null` values only when that is safe for the mapped property, while invalid required-field nulls are rejected instead of silently becoming defaults.
- Nullable type conversion preserves caller values when no converter exists and preserves explicit `null` results when a custom converter intentionally returns null.
- Backed enum defaults and backed enum scalar values are handled before date/default parsing, preventing enum defaults from tripping date checks.
- `EntityInspector` and `TypeResolver` can now be constructed directly; their singleton `getInstance()` methods are deprecated ahead of removal in 0.10.0.

## What Changed

### Bulk Insert Support

`EntityManager::insert()` can accept a list of entity-like rows:

```php
$manager->insert(ProductEntity::class, [
    ['name' => 'Desk', 'categoryId' => 10],
    ['name' => 'Chair', 'categoryId' => 10],
]);
```

Each row is validated as an entity-shaped payload. Malformed list entries or mixed numeric-key payloads are rejected before the ORM writes anything, avoiding accidental inserts of default or blank entities.

For generated primary keys, bulk insert results now include identifiers for rows created by the database. SQLite and MySQL read back generated IDs for full generated batches and mixed explicit/generated batches, while PostgreSQL continues to use `RETURNING` where available.

### Relation Alias Write Consistency

Relation ID aliases introduced in 0.9.4 now behave consistently across more write paths:

- update values can use relation aliases with scalar IDs or related objects;
- update readback conditions normalize relation aliases to their join columns;
- upsert conflict paths can use relation aliases;
- `UpdateOptions::writeNulls` accepts the mapped column name when clearing a relation alias;
- insert and upsert preserve declared scalar columns when their property name collides with a generated relation alias.

This keeps relation-backed foreign-key writes convenient without forcing duplicate scalar properties into the public entity shape.

### Safer Duplicate Column Writes

When multiple payload keys resolve to the same database column, the ORM now keeps column/value pairs aligned and compares duplicates strictly enough to preserve values such as `0`.

Matching duplicate writes are collapsed. Conflicting writes fail with a clear ORM error instead of silently choosing whichever value happened to be processed first.

### Partial Object Writes And Defaults

Partial object payloads are now more careful around nullable values and non-nullable mapped columns:

```php
$manager->update(ProductEntity::class, $dto, ['id' => 42]);
```

A nullable DTO property left as `null` can still mean "not changed" for PATCH-style updates. However, when a required mapped column has no real default and the caller supplies `null`, the ORM rejects the payload instead of silently retaining a PHP initializer and writing an unintended value.

Default handling also now accounts for backed enum defaults before date-format checks, so enum-backed columns can safely participate in partial create/update/upsert flows.

### Type Conversion

Type conversion now distinguishes between "no converter was found" and "a converter ran and returned null." This matters for nullable targets where a custom converter intentionally maps a value, such as an empty string, to `null`.

When no converter exists for a non-null supplied value, nullable target properties preserve the supplied value or surface the appropriate conversion error instead of silently dropping it to `null`.

### Singleton Deprecations

`EntityInspector` and `TypeResolver` now have public constructors. Their static `getInstance()` methods are deprecated and scheduled for removal in Assegai ORM 0.10.0.

New code should construct and inject these collaborators directly.

## Upgrade Notes

No database migration is required.

Bulk insert callers can continue using single-row inserts as before. If a caller previously passed malformed list-style arrays and relied on silent default hydration, those calls now fail validation and should be changed to pass a list of valid row payloads.

If an entity declares a real scalar property whose name matches a generated relation alias, the declared column wins. This preserves explicit entity mappings but may reveal payloads that were accidentally relying on the generated alias collision.

For partial object updates and upserts, use explicit null-write options when a `null` value is meant to clear a nullable column or relation. Required non-nullable fields without defaults should be omitted when unchanged, not supplied as `null`.

Replace new uses of `EntityInspector::getInstance()` and `TypeResolver::getInstance()` with direct construction or dependency injection before 0.10.0.

## Verification

- Focused EntityManager partial-write tests for relation aliases, null writes, duplicate column writes, enum/default handling, and type conversion
- SQLite bulk insert tests for generated and mixed explicit/generated identifiers
- MySQL integration coverage for bulk insert generated identifiers
